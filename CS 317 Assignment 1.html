Assignment Overview

In this assignment you will use the Java socket related classes to create an FTP client. Your program is to provide some basic FTP client functionality through a simple text-based interface. Only a subset of the FTP commands are to be implemented.

Your program will provide a simple command line interface to the user. The program will read a command typed by the user, and then interpret the command according to the command descriptions given below. Once the program has finished processing the command it will wait for another user command. Whenever the program is awaiting user input it is to print the text 'csftp> ' (Note the blank character after the >). It is to leave the cursor at that position awaiting input. (i.e., the program is not to print 'csftp>' on a line by itself. You may assume that command lines do not have more than 255 characters. Empty lines and lines starting with the character '#' are to be silently ignored, and a new prompt displayed.

The program is to take two command line arguments, the first is the FQDN or IP address of a host to connect to and the second is the port number to contact on the server. The port number is optional. If the port number is not supplied the program is to use port 21. If there are insufficient command line arguments then the program is to print a message and exit. The message provided in the starter code is sufficient. Assuming your program connects to the ftp server, the server will immediately send a response of either 120, 220, or 421. Upon receipt and processing of the response your program is to go into the above described command prompt mode. According to section 5.4 of the FTP RFC the communication between the client and server is an alternating dialogue. As a result your program can send a command and then wait for a response. There is no need to develop a multihreaded client.

If the connection fails, or is rejected, the program is to print the 0xFFFC error message. If the initial response from the server is incomplete, that is there is an I/O error then error 0xFFFD is printed.

A command line is composed of a command, as described above, followed by zero or more parameters. The command is separated from any parameters by one or more spaces and/or tabs. Tabs or spaces at the end of the line are to be ignored. Your FTP client may detect certain types of errors. When an error is detected the client is to print a single line consisting of the specified hex number and a short text message. In the list below, the message to print is in bold and it is followed by a description of the error it corresponds to, and if appropriate it describes how to proceed. Note that the client is only to exit when it gets the quit command or encounters one of the errors below that requires the client to quit.

  • 0x001 Invalid command. This is printed when the command entered by the user is not one of the accepted commands.
  • 0x002 Incorrect number of arguments. This is printed when the command is valid but the wrong number of arguments is provided. Note this could be the case if there are either too many or too few arguments.
  • 0x38E Access to local file XXX denied. In the message XXX is the file name of the local file that cannot be accessed. When performing a get it is possible that the client will not be allowed to create or write the named file. Observe that a get error on the remote side will result in response codes from the server indicating a problem and they will, through the normal course of processing by the client, be printed, so there is no need to print any additional information in that situation.
  • 0xFFFC Control connection to xxx on port yyy failed to open. When an attempt to establish the connection can't be completed within 20 seconds, or the socket cannot be created, then print this message, replacing xxx and yyy with the hostname, in the format on provided on the command line, and port number, in decimal, of the target ftp server you are trying to establish the control connection to. After printing the message the program is to exit.
  • 0xFFFD Control connection I/O error, closing control connection.If at any point an error while attempting to read from, or write to, the open control connection occurs, this message is to printed, and the socket closed/destroyed. The client is then to exit.
  • 0x3A2 Data transfer connection to xxx on port yyy failed to open. This message is to be printed when a data connection cannot be established within 10 seconds or the socket cannot be created. The xxx and yyy are to be replaced with the hostname, in the format provided on the command line, and port number, in decimal, of the target ftp server respectively.
  • 0x3A7 Data transfer connection I/O error, closing data connection. If at any point an error while attempting to read (or write, but that should never be the case) from the open data transfer connection occurs, this message is to printed, and the data transfer socket closed/destroyed. The client is then to go back to accepting commands. Note that this error does not trigger the closing of the command connection.
  • 0xFFFE Input error while reading commands, terminating. This error message is printed if an exception is thrown while the client is reading its commands (i.e., standard input). After printing this message the client will terminate.
  • 0xFFFF Processing error. yyyy. If for some reason you detect an error that isn't described above, print this message and replace yyyy with some appropriate text that briefly describes the error and exit

Your program will be run at the command line by doing java -jar CSftp.jar . (The supplied Makefile will also run the command if you enter make run)

All messages sent by the client to the FTP server over the control connection must be echoed to standard output with the prefix '--> ' (Note the space after the -->). All messages received on the control connection by the client program must be printed with the prefix '<-- ' (Note the space after the <--). Except for the text described above, no other text is to be printed by your program unless it is part of what the ftp client does or is listed as an error message your program is supposed to print.

To get started read RFC 959 to understand the nature of the messages exchanged between the client and server. (If you have access to the 6th edition section 2.3 of that edition also has an overview of FTP that might be helpful.) This site, http://www.cs.cf.ac.uk/Dave/Internet/node119.html, does a pretty good job of interpreting the ftp protocol and providing examples and explanations about what needs to be done.

Remember, You are only required to implement a subset of the protocol so some of the material in the references goes beyond what you need. Keep in mind that the RFC describes the data (protocol) exchanges between the ftp client (i.e., what you are writing) and an ftp server. It does not describe the commands typed by the user and accepted by the ftp client program, those are described in the table above.

Implementation Constraints and Suggestions

If you don't know how any of the ftp client commands above are supposed to work, check the 'ftp' command available on the department Linux servers. This program has all the commands you are required to implement. The -d command line option might be helpful in understanding what messages are exchanged between the client and server. Use some caution when interpreting the output with the -d option as the client issues a number of commands to the server that you do not have to implement. Some of these commands help optimize the interaction between the client and server but do not affect functionality. Just like the ftp client program on the lab machines, when your program starts it makes the connection prints the response from the server before printing the command prompt.

Don't spend too much time in the implementation of the shell. A simple show prompt, read line, identify command, run command is sufficient.

You aren't expected to implement any command that is not listed, but you can if you want.

Don't try to implement this assignment all at once. Instead incrementally build and simultaneously test the solution. A suggested strategy is to:

  • Read RFC to understand how the protocol works. Roughly the program works as follows
    • Establish a connection to a server
    • Send a user name
    • Transfer a file
  • Create a simple prompt, accept and parse simple commands but don't act on them
  • Read the arguments and create the connection based on the arguments;
  • Implement commands that do not require a data connection, like user, cd and quit;
  • Implement the get, and dir commands in passive mode;

Regarding the establishment of the data connection, your FTP client is to support passive mode only and it is to request that files transferred by the RETR ftp protocol command are to be in binary. Do not assume that the server will default to binary. It is okay to use binary mode for all transfers, including those used for the directory listing. In passive mode, before transferring a file, the client sends the PASV message and waits for a response. The response includes the port number the client is to establish a data connection on. Once the client establishes the data connection the file is transferred. If the ftp client tries to make a data connection to the server and the socket can't be created or a connection isn't established within a 10 seconds then the attempt to establish the connection should be abandoned, sockets etc closed/destroyed, and the 0x3A2 message printed.

For testing purposes you can connect to any FTP server, like ftp.cs.ubc.ca or ftp.debian.org (both accept the user 'anonymous' and any password). On some machines, but not our department machines, you might be able to run a server by issuing the command 'ftpd -p XXXX' where XXXX is a random port number greater than 1024 that you have selected. If supported, the command will start a new FTP server at the specified port. The login for this server is your account name and password, and the root directory is your home directory. Keep in mind that the password is sent in plain text and is not encrypted so if you connect to this ftp server over a network someone could sniff your password. (The department does have vsftpd that you can run, if you can figure it out.) Constantly typing in the ftp commands for your client can become a bit tedious after awhile. To reduce your typing you might want to checkout the command autoexpect on the cs machines. It basically lets you issue a series of commands, record the output and then play the commands back. You would start by doing: 

autoexpect java -jar CSftp.jar ftp.cs.ubc.ca 21
In the folder where CSftp.jar lives. You can then type in the sequence of commands that you wish to test and quit at the end. Once you're done autoexpect will generate a script.exp file that you can run by doing 
./script.exp
You can also edit the script.exp file to be more generous with respect to the expected responses.
